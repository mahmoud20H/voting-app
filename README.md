# Example Voting App

A simple distributed application running across multiple Docker containers.

## Getting started

Download [Docker Desktop](https://www.docker.com/products/docker-desktop) for Mac or Windows. [Docker Compose](https://docs.docker.com/compose) will be automatically installed. On Linux, make sure you have the latest version of [Compose](https://docs.docker.com/compose/install/).

This solution uses Python, Node.js, .NET, Redis for messaging, and Postgres for storage.

### Local deployment

The repository now includes a new authentication microservice and an Nginx reverse proxy.

1. Update the `.env` file in the repository root with the required secrets:

```env
REDIS_PASSWORD=your_redis_password
AUTH_SECRET=Generated by run.sh
DEFAULT_ADMIN_USER=admin
DEFAULT_ADMIN_PASSWORD=admin123
```

2. Start the stack from the repo root:

```shell
./run.sh
```

If you prefer non-interactive startup, use:

```shell
./run.sh --non-interactive
```

Alternatively, you can run Docker Compose directly after `.env` is present:

```shell
docker compose up -d --build
```

3. Open the app in your browser:

- Voting app: `http://localhost`
- Results dashboard: `http://localhost/results`
- Login page: `http://localhost/login`

### Authentication flow

- The `auth` service handles registration, login, JWT issuance, and token verification.
- The `vote` service requires users to authenticate before voting and redirects unauthenticated users to `/login`.
- The `result` service is protected behind an admin-only guard and is accessible through `/results` only for users with the `admin` role.
- Authentication state is stored in an HTTP-only cookie named `auth_token`.

### Service routing

Nginx proxies requests to the correct backend services:

- `/` → `vote`
- `/results/` → `result`
- `/auth/` → `auth`

### Environment and secrets

The stack requires the following environment variables in `.env`:

- `REDIS_PASSWORD`
- `AUTH_SECRET`
- `DEFAULT_ADMIN_USER`
- `DEFAULT_ADMIN_PASSWORD`


## Architecture

![Architecture diagram](architecture.excalidraw.png)

* A front-end web app in [Python](/vote) which lets you vote between two options
* An `auth` Python microservice that handles user registration, login, JWT token verification, and admin gating
* An Nginx reverse proxy that routes browser traffic to `vote`, `result`, and `auth`
* A [Redis](https://hub.docker.com/_/redis/) service which collects new votes
* A [.NET](/worker/) worker which consumes votes and stores them in…
* A [Postgres](https://hub.docker.com/_/postgres/) database backed by a Docker volume
* A [Node.js](/result) web app which shows the results of the voting in real time

## Notes

The voting application only accepts one vote per client browser. It does not register additional votes if a vote has already been submitted from a client.

This isn't an example of a properly architected perfectly designed distributed app... it's just a simple
example of the various types of pieces and languages you might see (queues, persistent data, etc), and how to
deal with them in Docker at a basic level.
